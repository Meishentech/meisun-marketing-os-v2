# V1 停用與完整搬遷到 V2 計畫

建立日期：2026-07-17

## 最新決策

使用者已確認：V1 現在可以朝「資料管理入口逐步停用」方向前進，未來所有行銷管理資料都應轉到 V2 管理，V2 最終取代 V1。

這不是立刻關掉 V1，而是代表後續批次的判斷原則改成：

- V2 做到同等或更好的管理能力後，V1 對應功能就應停用新增 / 編輯 / 刪除。
- V1 可以短期保留只讀查詢或轉址提示，避免使用者突然找不到舊資料。
- 所有後續新增生命週期都以 V2 為準，不再為了保留 V1 操作習慣而延後 V2 設計。

## 停用順序原則

1. **先補 V2 管理能力**
   在 V2 尚未能管理某張表前，不提早停用 V1 唯一可用入口，除非該表已被 V2 真實引用且 V1 刪除會破壞資料。

2. **再停 V1 寫入**
   每個模組搬完後，V1 對應的新增 / 編輯 / 刪除都要停用或改成提示。

3. **最後才處理 V1 顯示**
   V1 可先保留只讀，後續再決定是隱藏選單、顯示「已移至 V2」，或整站轉址。

4. **所有刪除改為 V2 軟取消 / 封存**
   V2 不恢復真刪除模式。任務 / 預算採取消語意，文件 / 資源 / 行銷案採封存語意。

## 建議批次

### Batch 13B：行銷案詳情三核心搬到 V2

目的：讓 V2 接手 v1 行銷案詳情頁最常用的任務、預算、文件。

正式規格：`BATCH13B_CAMPAIGN_DETAIL_SPEC.md`。

範圍：

- `marketing_campaign_tasks`：新增 / 編輯 / 軟取消。
- `marketing_campaign_budget_items`：新增 / 編輯 / 軟取消。
- `marketing_campaign_documents`：上傳 / 編輯 / 封存。
- 採用「詳情頁 + 巡檢卡片」混合架構：詳情頁深耕單一案子，頂部巡檢卡片進入跨案只讀列表。
- 行銷案詳情頁手機優先，任務 / 預算 / 文件要能在手機完整操作，巡檢列表點列後回到詳情頁編輯。
- 已取消 / 已封存資料提供只讀歷史清單，Phase 1 不做恢復。

決策沿用 Batch 13A Claude 審查：

- tasks / budget items 用 `cancelled_at`、`cancelled_by`、`cancel_reason`。
- documents 用 `archived_at`、`archived_by`、`archive_reason`。
- `all_expenses_overview` 排除已取消預算項目。
- 文件版本規則已拍板：新版本 = 新增一筆文件，不覆蓋舊檔；封存套用在單筆文件。
- 即將到期任務：未完成、未取消、逾期或 7 天內到期。
- 待付款項目：未取消、付款狀態不是已付款 / 不需付款，且有有效金額。

Batch 13B 完成後，V1 應同步：

- 停用 `delTask()`、`delBudgetItem()`。
- 文件刪除已於 Batch 13A 停用，需確認仍維持。
- 評估是否停用 V1 任務 / 預算 / 文件新增與編輯，或至少加提示「請至 V2 管理」。

### Batch 14：風險 / 待決策搬到 V2

目的：整合 `marketing_campaign_risks`、`marketing_campaign_risk_updates` 與 V2 總經理待決策中心。

範圍：

- V2 顯示與管理行銷案風險。
- 風險追蹤更新可新增 / 編輯 / 軟取消。
- 重要風險可進總經理待決策中心。

完成後 V1 應同步：

- 停用風險與追蹤紀錄的新增 / 編輯 / 刪除。

### Batch 15：行銷成效 / Channel 成效搬到 V2

目的：V2 接手 `marketing_campaign_performance`，讓總經理與行銷總監看到真實 Channel 成效。

範圍：

- 行銷總監可新增 / 編輯成效資料。
- 總經理看摘要、趨勢、轉換率與待改善 Channel。
- 業務只看與素材 / 商機支援有關的結果。

完成後 V1 應同步：

- 停用成效資料新增 / 編輯 / 刪除。

### Batch 16：公會管理完整搬到 V2

目的：把 V1 公會 CRUD 搬到 V2，並整合 V2 已建立的標籤、階段與合作 overview。

範圍：

- 公會主檔、年費、權益、任務、任務費用、期刊、活動、備註。
- 原則上全部採軟取消或封存，不再真刪除。

完成後 V1 應同步：

- 停用公會系列所有新增 / 編輯 / 刪除。

### Batch 17：投標工具完整搬到 V2

目的：讓行銷總監在 V2 管理標案監測，業務在 V2 使用標案結果。

範圍：

- `tender_projects`、`tender_keywords` 的新增 / 編輯 / 停用。
- `tender_results` 狀態更新與轉名單流程。
- 掃描排程可先留在舊 repo 或獨立腳本，前端操作移到 V2。

完成後 V1 應同步：

- 停用投標工具管理入口，或改為轉到 V2。

### Batch 18：內容營運功能搬到 V2

目的：搬遷較獨立的內容功能。

建議順序：

1. 成功案例庫。
2. 新聞關鍵字與新聞蒐集。
3. 每週文案草稿與排程管理。

### Batch 19：V1 全站只讀 / 停用

目的：V2 已完整接手後，把 V1 轉成只讀、公告或導向 V2。

可選方案：

- **保守方案**：V1 保留登入與只讀查詢，但所有管理按鈕停用。
- **明確轉移方案**：V1 登入後顯示「本平台已移至 V2」，附 V2 連結。
- **最終方案**：Cloudflare Pages 將主要路徑轉址到 V2，只保留必要備份路徑。
