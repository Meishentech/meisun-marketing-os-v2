## 阻塞問題

### 1.（已用瀏覽器實測確認）登入畫面的品牌名稱文字是白色，在白色卡片背景上完全看不見

實際截圖看到「MS」標記旁邊只清楚顯示「V2 SECURE ACCESS」，上面應該要有的「美昇 Marketing OS」整行文字幾乎是隱形的（只剩極淡的殘影）。用 `getComputedStyle` 追出原因：

```html
<div class="brand login-brand">
  <div class="brand-mark">MS</div>
  <div>
    <strong>美昇 Marketing OS</strong>   <!-- color: rgb(255,255,255)，白色 -->
    <span>v2 secure access</span>        <!-- color: rgb(159,196,223)，可正常辨識 -->
  </div>
</div>
```

**根本原因是 CSS 選擇器優先權/順序問題，不是這次手機優化 commit 造成的**：`styles.css` 裡同時存在 `.login-brand strong { color: var(--text); }`（第 61 行，給登入卡片這種淺色背景用）跟 `.brand strong { color: #fff; }`（第 153 行，給側邊欄那種深色背景用）。這個元素的 class 是 `"brand login-brand"`，兩條規則的優先權完全相同（都是「一個 class + 一個元素」），**CSS 規則優先權相同時後面寫的贏**——`.brand strong` 寫在後面，所以白色蓋掉了原本應該生效的深色，導致登入頁的品牌文字在淺色卡片背景上變成隱形。

**這個 bug 在桌機寬度下也存在**（已實測確認，不是手機特有問題），只是這次因為你們把「手機必須能完整查看」訂為明確的產品原則，這個「內容完全看不到」的問題被放進這次審查範圍。**建議列為阻塞問題，理由是它直接違反新訂的原則，而且是每個使用者登入時第一眼看到的畫面**，即使功能上（Email/密碼欄位、登入按鈕）都還能正常操作。

**建議修法**：把 `.login-brand strong`/`.login-brand span` 這兩條規則搬到 `.brand strong`/`.brand span` **後面**（利用「後寫的贏」解決，不用改任何顏色值），或是把選擇器改成更明確的 `.login-card .login-brand strong`（提高優先權，不用管寫在哪裡）。兩種都是幾分鐘內能改完的小修正。

## 建議立即修正

### 2.（讀原始碼推算，未實測）廠商合作/交付物頁：手機版一個廠商卡片裡塞了巢狀的交付物清單跟按鈕，操作範圍容易混淆

這是這次審查在「表格轉卡片」的通用機制之外，特別把 Batch 6C（`a8043a5`）建的廠商合作頁拿出來重新檢查發現的問題。`formatDeliverableSummary()` 目前的做法是把「這個廠商底下最多 3 筆交付物、每筆各自的名稱/狀態/日期文字、以及各自的『編輯』『取消』按鈕」全部串成一個字串塞進「交付物」這一欄的儲存格，用 `<br>` 換行分隔。

在桌機版，這欄只是一整排欄位裡的其中一欄，旁邊有其他欄位框住視覺範圍，還算看得出「這些是同一列裡的子清單」。但手機版套用這次新加的表格轉卡片機制後（`.table td { display:grid; grid-template-columns: minmax(92px,34%) minmax(0,1fr); }`），每一欄都變成獨立的一個標籤+內容區塊，垂直往下疊——「交付物」這一格會變成一長串內容：3 筆交付物的文字說明，每筆後面各跟著自己的「編輯」「取消」按鈕，緊接著下面又是這個**廠商本身**的操作按鈕列（`vendorActionGroup` 的「編輯」「交付物」「送審」「取消」）。

**問題**：交付物層級的「編輯」「取消」按鈕跟廠商層級的「編輯」「取消」按鈕文字完全一樣，在手機版被壓縮到同一欄寬度、上下緊鄰排列時，使用者很容易分不清楚點到的「取消」是要取消某一筆交付物，還是要取消整個廠商合作——這剛好就是新訂的產品原則特別強調的「手機上必須能完整...取消」，如果取消的對象認錯了，後果不是小事。

**建議立即修正（低成本）**：至少先在視覺上把交付物子清單跟外層的廠商操作按鈕明確分開——例如幫交付物子清單的區塊加上淺色背景框或左側縮排線（跟現有 `.mini-card`/`.list-item` 的卡中卡視覺語言一致），讓使用者一眼看出「這一段按鈕是屬於某筆交付物，不是屬於整個廠商」。這個修正只動 CSS，不用重新設計資料流程。

**建議之後再做的正確解法（第「可延後優化」節）**：把交付物子清單從表格儲存格裡搬出來，改成廠商卡片下面的獨立子區塊或另開一個小 modal 管理，而不是塞進一格文字裡——這個做法在桌機版其實也會比較清楚，不是手機專屬的修正，但工作量比第一步大，不建議現在為了這次手機優化順便做。

## 可延後優化

1. **交付物清單改成獨立子區塊或 modal**（上面第 2 點的長期正確解法）——不急，先做視覺分區的小修正即可，這個之後再排。
2. **Modal 底部按鈕順序**：手機版 modal 的主要動作按鈕（例如「確認刪除」「建立」）目前排在上面、次要動作（「取消」）排在下面（`.modal-actions .secondary-button { order: 2 }`）。這不是錯誤，只是跟部分手機系統慣例（次要動作在上、主要動作貼近畫面底部）不完全一致，屬於見仁見智的排版選擇，不影響功能，不急著改，等之後有使用者實際回饋再考慮調整順序。
3. **只有單一斷點（760px）**：目前手機版樣式只用 `@media (max-width: 760px)` 一組規則，沒有另外再細分「手機」跟「小平板」。以現在的資料密度（3-7 欄的表格、KPI 卡片）來看，用一組斷點應該還撐得住，等實際在更小的手機（例如 iPhone SE 這種 375px 寬度）上跑過一輪、如果真的有哪個頁面在最窄的螢幕上還是太擠，再考慮加一個更窄的斷點，不用現在預先做。

## 是否建議進入下一批功能

**可以繼續，但建議先把「阻塞問題」那一項（登入畫面白字）修掉，因為修正成本幾分鐘、影響範圍是每個人登入時的第一印象。** 「建議立即修正」那項（廠商合作頁按鈕範圍混淆）不會擋住其他批次的開發，但因為牽涉到「點錯會取消錯東西」這種操作風險，建議跟下一批一起排進去，不要無限期往後延。

## 各檢查項目簡答

1. **三個角色在手機寬度下是否可用**：沒有登入憑證，無法實際測試三個角色的畫面，這部分是讀原始碼確認 CSS 規則的涵蓋範圍（KPI 卡片、表單、按鈕、彈窗都有對應的手機樣式），沒有發現遺漏的角色專屬頁面類型，但建議之後找機會用真實帳號實際跑一遍三個角色確認。
2. **表格轉卡片後欄位名稱/內容/按鈕是否清楚**：機制本身（`data-label` + CSS `content: attr()`）做法正確，「年度行銷案彙總」「費用狀態」這兩張表結構單純（欄少列少或欄多列少），轉卡片後應該清楚；「廠商合作/交付物」因為儲存格裡塞了巢狀清單跟按鈕，是唯一有實質疑慮的地方（見上方第 2 點）。
3. **廠商合作/交付物、業務需求單、年度預算總表、待決策中心的手機操作風險**：業務需求單（單純表格、每列最多兩顆按鈕）、年度預算總表（少列多欄，很適合轉卡片）、待決策中心（原本就是用 CSS Grid 的清單/卡片型態，不是表格，這次沒有改動也不需要改動）三個沒有發現風險；廠商合作/交付物是唯一需要處理的（見上方）。
4. **Modal 表單在手機上是否容易填寫送出**：`.modal-body` 本來就有 `overflow:auto`，這次只是調整手機版的高度計算（`calc(100vh - 94px)`），長表單（例如新增廠商合作那個有 10 個欄位的表單）在手機上應該可以正常捲動，不會被截斷；輸入框/選單也都有 `min-height: 46px`，觸控目標大小合理。
5. **文字溢出/按鈕太小/需要橫向拖曳/內容層級混亂**：`.campaign-summary-table` 有正確把桌機版的 `white-space: nowrap` 在手機版改回 `normal`，避免文字被裁切；行動裝置的按鈕 `min-height: 36-44px` 符合一般建議的觸控大小；沒有發現需要橫向捲動的地方（表格已經全部轉成直向卡片）；「內容層級混亂」正是第 2 點提到的廠商合作頁問題。
6. **是否影響桌機版**：這次的異動全部包在 `@media (max-width: 760px)` 裡面，`data-label` 屬性本身在桌機版沒有對應的 CSS 規則所以不會顯示，桌機版視覺應該完全不受這次 commit 影響。但要注意「阻塞問題」提到的登入白字 bug，本身不在這次 commit 範圍內、也不是手機限定，桌機版一樣看不到那行字，修的時候會同時修好兩種寬度。
