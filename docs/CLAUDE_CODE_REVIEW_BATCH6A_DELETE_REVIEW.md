# Claude Code Review｜Batch 6A 複查：業務需求單刪除功能

審查日期：2026-07-15
審查對象：commit `3c0b550`（`sales_requests` 刪除功能、移除 `dataStatusText()` 提示）
審查方式：實際讀取 `assets/app.js` 目前的 `openDeleteSalesRequestModal`、`visibleSalesRequests`、共用 modal 送出邏輯（`modalSubmitHandler`）。

---

## 是否通過

**不通過，建議改成「取消」而非真刪除，再上線。** 前端實作本身（modal 確認、送出鎖定、錯誤訊息、重新讀取）品質良好、沒有明顯的程式 bug；問題出在資料模型層——真 `DELETE` 會讓這個模組原本設計要支撐的「需求量、平均處理天數、業務支援紀錄」這幾個指標出現無法回補的資料缺口，而且刪除動作沒有記錄「誰刪的、為什麼刪」，這對一個本來就是設計來**留存業務跟行銷之間往來紀錄**的模組來說，是核心用途被削弱，不只是錦上添花的小事。

移除 `dataStatusText()` 那部分沒有問題，那是開發期的除錯提示，功能穩定後拿掉是合理的收尾動作，不影響任何資料，這部分審查通過。

---

## 1. 需求單用真刪除是否合理，還是應改成「取消 / 封存」保留歷史

**應改成取消，不建議真刪除。** 理由：

- 這個模組存在的目的之一就是讓行銷總監能看到「業務需求量」「平均處理天數」這類流程指標（原始 wireframe 就有「平均回覆：2.5天」這個 KPI），這些數字必須靠**完整**的歷史紀錄才能算，真刪除會讓統計基礎不完整，而且沒有任何方式能事後判斷「這裡本來有多少筆被刪掉了」。
- 最risky 的情境是刪除**已經完成**的需求單——這種需求單通常已經關聯了實際產出（`deliverable_resource_id` 指向真的 `marketing_resources`），一旦刪除，就永久失去「這份文宣/簡報當初是為了回應哪個業務需求而做的」這個追溯關聯，而目前的實作對「這筆是待處理」跟「這筆已完成」一視同仁，同一顆刪除鈕、同一句確認文案。
- 這不只是使用者不小心點錯的風險（單一確認彈窗本來就沒辦法完全防呆），更根本的是：即使是**刻意、正確**的刪除動作，事後想回頭查「上個月那筆需求後來怎麼了」也會完全查不到。

---

## 2. 業務端是否應限制只能刪自己的需求，行銷總監是否能刪所有需求

**目前的 UI 層行為已經是對的**，不用改：`visibleSalesRequests()`（第 1081-1086 行）已經確認業務只會看到自己的需求單（`return ownRequests`，之前 fallback 顯示所有人的邏輯已經拿掉），所以業務畫面上的「刪除」鈕實際上只會出現在自己的需求列旁邊；行銷總監因為 `isMarketing` 分支回傳全部資料，所以能對任何一筆按刪除——這對應行銷總監本來就該有的管理權限，設計上是對的。

**但這只是 UI 層的限制，不是真正的存取控制**：目前 `sales_requests` 的 grant 是任何登入帳號都能 `delete`（Batch 4 建表時就是這樣設的），所以業務理論上可以繞過畫面，直接呼叫 `DELETE /rest/v1/sales_requests?id=eq.{別人的id}`，畫面上看不到不代表 API 層擋得住。這點跟第 3 節一起討論。

---

## 3. 目前 Phase 1 只靠前端控制，`sales_requests` 的 RLS 是否需要提前加強

**如果採用第 4 節的「改成取消」建議，這個風險會自動下降一個等級，不需要額外加 RLS。** 原本「真刪除 + 沒有列層級保護」的組合，最壞情況是資料被永久抹除且查不到是誰做的；改成「取消」之後，就算真的有人繞過 UI 用 API 把不該取消的需求標成取消，那筆紀錄還在（`cancelled_by`/`cancelled_at` 會留下痕跡），是可以事後發現、可以復原（把 `cancelled_at` 清回 `null` 就好）的動作，風險性質從「不可逆的資料損失」降級成「可追蹤、可回復的誤操作」。

這跟 Batch 6A 草案審查當時對 `approval_requests` 提過的判斷邏輯是一致的：**寫入動作要不要提前加 RLS，取決於這個動作可不可逆、有沒有留下痕跡**，不是每個寫入動作都要用同一套標準。改成取消之後，`sales_requests` 這裡可以維持 Phase 1 現有的「前端控制、資料庫層晚點再做」判斷，不需要為了這個功能特別提前處理 RLS。

## 4. 未來要統計平均處理天數、需求量、業務支援紀錄，DELETE 會不會造成資料缺口

**會，而且是這次審查不通過的核心理由。** 已完成需求被刪除後：

- `需求總數`（`requestKpis()` 裡的 `total`）會少算，且沒有任何方式能回補正確的歷史總數。
- `平均回覆天數`（`created_at` 到 `completed_at` 的差）如果被刪的剛好是處理得比較慢或比較快的案例，會讓平均值產生偏差，而且這個偏差是**看不見的**——沒有人會知道統計數字已經不準了。
- 「業務支援紀錄」這種需要回頭查「某個業務過去提過哪些需求、行銷團隊怎麼支援他」的情境，一旦中間有任何一筆被刪，這條時間線就會出現斷點。

改成取消（軟刪除）之後，這些指標的查詢只要固定加上「排除 `cancelled_at is not null` 的列」就能維持跟現在完全一樣的行為（畫面上看到的數字不會變），但歷史資料完整保留，之後真的要做「處理天數」「需求量趨勢」這類報表時，資料庫裡有完整的底可以查，不用擔心中間有無法解釋的缺口。

## 5. 目前前端實作是否有 bug 或 UI 風險

**沒有發現程式錯誤，實作品質本身是好的**，逐項確認：

- 共用 modal 的送出邏輯（第 2134-2152 行）在呼叫 `modalSubmitHandler` 前會 `submit.disabled = true`，失敗時用 `try/catch` 顯示錯誤訊息、`finally` 恢復按鈕，**沒有雙擊送出兩次刪除請求的風險**。
- 點擊 modal 背景（`formModal` 本身）只會關閉視窗，不會誤觸發 `onSubmit`，**不會意外刪除**。
- 刪除後呼叫 `loadExistingData()` 重新整理，空列表的三層判斷（有資料/已連線但沒資料/未連線示範資料）在更早的修正裡已經處理過，刪光之後畫面會正確顯示「你目前沒有自己提出的需求單」，不會出現殘留的假資料或報錯畫面。

**唯一的軟性建議（不是 bug）**：確認訊息目前對業務刪自己的需求跟行銷總監刪別人的需求用的是同一句文案（「確定要刪除『{需求名稱}』嗎？」），沒有顯示提出人是誰。行銷總監如果同時管理很多業務的需求單，看到需求名稱不一定能馬上想起是誰提的——如果改成「取消」，順手把提出人也放進確認文案裡（「確定要取消業務 A 提出的『{需求名稱}』嗎？」），對行銷總監操作時的辨識度會更好，這個小改動可以跟第 6 節的調整一起做。

## 建議：改成取消（soft cancel），最小 SQL 與前端調整方向

### 最小 SQL

```sql
alter table sales_requests
  add column if not exists cancelled_at timestamptz;

alter table sales_requests
  add column if not exists cancelled_by citext references app_user_access(email);

create index if not exists idx_sales_requests_cancelled
  on sales_requests(cancelled_at);
```

兩個都是 nullable 純新增欄位，不影響既有資料，不需要動任何既有查詢的欄位定義，也不用碰 `status` 這個既有欄位的語意（`status` 繼續代表「處理到哪個階段」，取消是獨立於階段之外的另一個維度，這樣事後還能看出「這筆是在處理中被取消的，還是連還沒開始處理就被取消的」）。

### 前端調整方向

1. **`openDeleteSalesRequestModal` 改成 `openCancelSalesRequestModal`**：確認文案改成「取消」語氣（例如「確定要取消這筆需求單嗎？取消後會保留紀錄，但不再出現在待處理清單。」），送出時改成：
   ```js
   await api("PATCH", `sales_requests?id=eq.${encodeURIComponent(id)}`, {
     cancelled_at: new Date().toISOString(),
     cancelled_by: state.auth.email,
     updated_at: new Date().toISOString(),
   });
   ```
   不再呼叫 `DELETE`。
2. **按鈕文案**：「刪除」改成「取消」，`is-danger` 樣式可以維持（取消仍然是需要提醒的動作，紅色語氣沒有問題）。
3. **既有的讀取查詢加上排除條件**：`loadExistingData()` 裡讀 `sales_requests` 的 `safeGET` 呼叫，加上 `&cancelled_at=is.null`，這樣被取消的需求單不會出現在正常列表跟 KPI 計算裡，畫面行為維持跟現在一樣（不會多一堆「已取消」項目干擾業務/行銷總監日常操作）。
4. **（可選，非必要）之後如果想讓行銷總監能查「哪些需求被取消過」**，可以另外開一個小的篩選或列表，查詢時不加 `cancelled_at=is.null` 這個條件即可看到全部歷史，這部分不急著在這批做，先確保資料庫裡有留底就好。
5. 第 5 節提到的確認文案加上提出人姓名，可以一起做。

---

## 是否阻塞下一批功能

**不阻塞。** 這個調整範圍很小（兩個 nullable 欄位 + 一個函式從 DELETE 改成 PATCH + 讀取查詢加一個過濾條件），可以在下一個小批次或現在直接補上，不需要重新設計其他已經完成的批次。建議在繼續往下一個功能項目前，先把這個改掉——不是因為技術上會卡住後面的功能，而是因為**目前上線的版本，只要有人點過一次「刪除」，就會產生一個之後補救不了的資料缺口**，越晚改，已經被真刪除、回不來的資料就可能越多。
