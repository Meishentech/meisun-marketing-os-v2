# V2 Production Smoke Test

建立日期：2026-07-20

用途：每次 V2 production 部署、live Supabase SQL migration、或 V1/V2 邊界調整後，照本文件快速確認核心流程沒有壞掉。

## 測試原則

- 測試站台：`https://meisun-marketing-os-v2.pages.dev`
- 只測 V2，不回 V1 建立或修改核心行銷資料。
- 測試資料名稱請加前綴：`SMOKE YYYY-MM-DD`，方便後續辨識。
- 有生命週期的資料用「取消 / 封存」收尾，不做真刪除。
- SQL migration 不能只看檔案或 commit，必須確認 live Supabase smoke test 有回傳結果。
- 若 production 顯示 live 空資料，應顯示空狀態，不可退回 demo 假資料。

## 測試前確認

1. GitHub 最新 commit 已部署到 Cloudflare Pages production。
2. 若本批有 SQL，已在 live Supabase SQL Editor 執行。
3. SQL smoke test 有實際回傳欄位或筆數，不是只有「Success. No rows returned」。
4. 測試帳號可登入，且帳號在 `app_user_access` 為啟用狀態。
5. 手機驗收請至少用 375 x 812 寬度跑一次核心流程。

## 登入與角色

### Admin / Eric 視角切換

步驟：

1. 使用具 admin 權限的帳號登入。
2. 確認左側或頂部可切換「總經理 / 行銷總監 / 業務」視角。
3. 分別切換三個視角。

成功標準：

- 三個視角都能載入。
- 切換後頁面選單符合角色。
- 畫面不顯示「目前已接 Supabase」這類開發提示文案。

### 一般業務

步驟：

1. 使用一般業務帳號登入。
2. 檢查是否只看到業務視角。

成功標準：

- 不能切換到總經理或行銷總監。
- 業務需求單只顯示自己的需求。

## 總經理

### 戰情室

步驟：

1. 進入「總經理戰情室」。
2. 檢查年度預算總表、上下半年專案表、重要性與執行狀態顏色。
3. 點進「商機轉換」「Channel 成效」「待決策中心」「週報摘要」。

成功標準：

- 年度預算表有上半年 / 下半年 / 全年度統整。
- 高重要性使用明顯警示色，能快速判斷急迫性。
- 若 live 沒有資料，顯示清楚空狀態，不顯示 demo 案例。

### 待決策中心

步驟：

1. 進入「待決策中心」。
2. 若有待審核資料，開啟一筆審核。
3. 若沒有待審核資料，確認空狀態。

成功標準：

- 空資料時不顯示假待審核項目。
- 審核 modal 可完整顯示內容與按鈕。
- 核准 / 駁回後狀態會更新，且不會卡在 loading。

### Channel 成效

步驟：

1. 進入「Channel 成效」。
2. 檢查 Channel 列表是否讀取 `marketing_campaign_performance.channel` 與 `leads.source_channel`。
3. 找一個有 performance 資料的 Channel，以及一個只有 leads 補缺的 Channel。

成功標準：

- 同一 Channel 有 performance 紀錄時，詢問、名單、有效名單以 performance 為主。
- 只有 performance 缺資料時，才用 leads 來源補缺。
- 不出現 performance + leads 重複疊加造成的膨脹數字。

### 週報摘要

步驟：

1. 進入「週報摘要」。
2. 檢查本週期間、專案進度、風險 / 待決策、付款、Channel 段落。
3. 點「複製週報」。
4. 點「匯出 .txt」。

成功標準：

- 週報文字可完整閱讀。
- 複製成功或有可手動複製的 fallback。
- 匯出的 `.txt` 可開啟。
- 週報只讀，不自動新增或修改 `approval_requests`。

## 行銷總監

### 行銷專案管理

步驟：

1. 進入「行銷專案管理」。
2. 建立一筆 `SMOKE YYYY-MM-DD 行銷案`。
3. 編輯該行銷案，修改狀態、重要性、預算與期間。
4. 進入專案詳情。
5. 封存該行銷案。

成功標準：

- 新增後出現在列表。
- 編輯後資料立即更新。
- 封存後移到「已封存行銷案」清單。
- 封存案不出現在新增廠商合作的專案下拉選單。

### 任務

步驟：

1. 在測試行銷案詳情新增任務。
2. 編輯任務的負責人、日期、狀態與完成度。
3. 取消任務並填寫原因。
4. 展開已取消任務清單。

成功標準：

- 新增 / 編輯可儲存。
- 取消後從進行中任務列表移除。
- 已取消清單保留取消人、取消時間與原因。

### 預算 / 付款

步驟：

1. 新增預算項目，填寫台幣金額、報價狀態、付款狀態與付款日。
2. 編輯付款狀態。
3. 取消預算項目。
4. 檢查總經理支出 / 待付款摘要是否排除已取消項目。

成功標準：

- 台幣 / 人民幣欄位比例與欄寬在手機上正常。
- 取消後不再污染待付款巡檢與支出彙總。
- 已取消清單可查到歷史原因。

### 文件

步驟：

1. 新增一份文件版本並上傳小型測試檔。
2. 點「開啟」確認 signed URL 可下載或預覽。
3. 編輯文件中繼資料，不換檔。
4. 新增另一個版本。
5. 封存其中一份文件。

成功標準：

- 私有 Storage 檔案不以原始 `file_path` 直接開啟，而是透過 signed URL。
- 編輯文件不會覆蓋舊檔。
- 新版本是新增一筆資料。
- 封存文件移到已封存清單。

### 風險 / 待決事項

步驟：

1. 新增一筆高影響風險。
2. 編輯負責人、狀態、到期日。
3. 新增追蹤紀錄。
4. 編輯追蹤紀錄。
5. 取消追蹤紀錄。
6. 封存風險。

成功標準：

- 風險負責人可編輯，不被鎖成登入者。
- 高風險 / 逾期項目會進入總經理與行銷總監摘要。
- 已取消追蹤不影響最新追蹤與逾期判斷。
- 已封存風險保留歷史追蹤。

### 成效資料

步驟：

1. 在行銷案詳情新增成效資料。
2. 填寫 Channel、觸及、名單、詢問、有效商機、成交件數與成交金額。
3. 再次開啟同一行銷案成效並編輯。
4. 回到 Channel 成效頁檢查數字。

成功標準：

- 同一行銷案只能有一筆成效資料。
- 若遇到 unique constraint，畫面需顯示中文提示並重新載入資料。
- Channel 成效頁讀到最新數字。

### 合作廠商 / 交付物

步驟：

1. 新增廠商合作，選擇未封存行銷案。
2. 新增交付物。
3. 編輯廠商合作與交付物。
4. 取消交付物。
5. 取消廠商合作。
6. 展開已取消清單。

成功標準：

- 新建廠商後若第二段寫入失敗，重試不會重複建立孤兒廠商。
- 取消交付物不會取消整個廠商合作。
- 取消廠商合作後，相關待審核報價只處理待審核狀態，不動已核准歷史。
- 已取消清單能看到類型、時間、原因。

### 文宣資源管理

步驟：

1. 新增一筆 `SMOKE YYYY-MM-DD 文宣資源`。
2. 上傳小型測試檔或填入外部連結 / Canva 連結。
3. 編輯資源內容。
4. 封存資源。
5. 到業務端確認封存資源不再顯示。

成功標準：

- 私有檔案下載會產生 signed URL。
- 封存前顯示引用數量。
- 已封存但仍被知識庫引用的資源，在知識條目中顯示「已封存」且不可下載。

### 產品知識庫

步驟：

1. 新增或編輯一筆知識條目。
2. 填寫詳細說明、建議話術、不建議話術、競品對照。
3. 連結一筆文宣資源。
4. 移除資源連結。

成功標準：

- 知識類型下拉不會把既有值靜默覆蓋成第一個選項。
- 業務角色只看得到可對外 / 僅內部允許的內容。
- 移除連結只刪除 `product_knowledge_resource_links`，不刪原始資源。

### 業務需求單

步驟：

1. 以業務角色建立需求單。
2. 以行銷總監角色查看需求單。
3. 更新狀態。
4. 取消需求單。

成功標準：

- 業務只看到自己的需求。
- 行銷總監看到全部需求。
- 取消是 soft cancel，不是真刪除。
- 已取消清單可展開查看。

## 業務

### 文宣 / 資源下載

步驟：

1. 進入「文宣 / 資源下載」。
2. 開啟外部連結、Canva 連結。
3. 下載可對外使用的私有檔案。
4. 確認封存或不可下載的資源不顯示下載動作。

成功標準：

- 私有檔案下載成功產生 signed URL。
- 內部不可落地下載的資料不會給下載按鈕。
- 手機上多個按鈕可清楚堆疊，不擠壓。

### 產品知識庫

步驟：

1. 進入「產品知識庫」。
2. 開啟一筆知識條目詳情。
3. 點「提出補充需求」。

成功標準：

- 不顯示禁止使用或待確認條目。
- 業務看不到「編輯條目」「新增資源連結」「移除連結」。
- 補充需求會預填知識條目資訊。

### 業務需求單

步驟：

1. 建立一筆需求單。
2. 查看自己的需求列表。
3. 取消一筆需求。
4. 展開已取消需求。

成功標準：

- 不會看到其他業務的需求單。
- 取消後不在進行中列表。
- 已取消資料保留建立人與取消時間。

## 手機版必測

每次 production smoke test 至少測以下手機寬度：

- 375 x 812

必測項目：

- 登入頁品牌文字可讀。
- 總經理戰情室年度預算表不橫向爆版。
- 行銷案詳情頁可操作任務 / 預算 / 文件 / 風險 / 成效 modal。
- Modal 底部按鈕可觸達。
- 合作廠商交付物卡片的「編輯 / 取消」歸屬清楚。
- 週報摘要文字區可讀、可捲動、可複製。
- 業務資源多按鈕情境不重疊。

## 防回歸檢查

### V1 / V2 邊界

1. V1 對 8 張核心表的新增 / 編輯入口保持凍結。
2. V1 儲存函式直接呼叫時只顯示「此功能已移至 V2，請至新版行銷管理平台操作。」。
3. V1 可保留列表、下載、匯出，但核心資料子項詳情與操作以 V2 為準。

### Demo 資料

1. live Supabase 已連線時，不可顯示舊 demo row。
2. 空資料用空狀態，不用假資料補版面。

### SQL / Schema

1. 若前端出現 `PGRST204`，先檢查 live 欄位是否已建立。
2. 若新表出現 `relation does not exist`，先確認 SQL 是否真的跑到 live。
3. 若 view 讀取失敗，檢查是否有 `grant select ... to authenticated` 與 `security_invoker = true`。

## 收尾紀錄格式

每次跑完 smoke test，請在對應交接文件或 PR / commit 記錄中留下：

- 測試日期。
- 測試站台。
- 測試帳號角色。
- 有無新增測試資料。
- SQL smoke test 結果。
- 手機寬度是否測過。
- 阻塞問題與修正 commit。
