## 1. 這三個操作是否是最適合優先做的 Batch 6A 範圍

**操作 1（業務提出需求）跟操作 2（行銷總監更新需求）是對的優先順序**——這兩個是「建立 → 處理」完整的一條線，做完之後業務需求單這個模組是真的能運作的，不會半殘。

**操作 3（總經理核准/需修正）目前缺少建立資料的一端**：重新檢查整個 repo，`approval_requests` 從 Batch 4 建表到現在，沒有任何程式碼寫入過一筆真實資料（Batch 4 驗收時提到的「已新增測試需求」指的是 `sales_requests` 那張表，不是 `approval_requests`）。原本 Batch 3/4 審查時就提過「等 `approval_requests` 建好後，建議把廠商報價待核准接進去」，但這個串接到現在還沒有人做，而且**廠商合作頁（`vendorSection`）本身也還是唯讀的**，連行銷總監要把某筆報價標成「待核准」的介面都不存在。

**建議三選一，我的建議是選項 A：**

- **A（建議）**：Batch 6A 範圍縮小成操作 1 + 操作 2 + 操作 4（重新讀取），操作 3 的核准/需修正 UI 一起做沒問題（程式碼不難），但**額外加一個小動作**：行銷總監編輯廠商合作報價狀態時（`marketing_campaign_vendors.quote_status` 改成「待核准」），順手 `insert` 一筆 `approval_requests`（`entity_type='vendor_quote'`，欄位對照第 3 節）。這樣總經理那頁做完之後至少有一條完整能跑的路徑可以驗收，不是空殼功能。這比額外開一個 Batch 6B 專門處理創建端更有效率，因為欄位/串接規則在 Batch 3/4 就已經定案了，不需要重新設計。
- **B**：維持原範圍三項都做，但明確接受「總經理那頁短期內只能靠手動塞測試資料驗收，等之後某一批做了廠商報價編輯或預算核准送審才會有真實資料」——不是不行，只是要讓總經理知道這個功能現在還看不到真實案例。
- **C**：把操作 3 整個挪到下一批，Batch 6A 只做操作 1 + 2 + 4，等有明確的「送審」入口規劃好再一起做操作 3。

這件事需要你決定要選哪個，不建議我自己代為縮減或擴大範圍（第 7 節列為待確認事項）。

## 2. 前端應該用 Modal、Inline Row Action，還是獨立詳情頁

**建議用 Modal，理由跟現有架構直接相關，不是憑一般偏好判斷：**

檢查 `render()`/`renderSections()` 現況（`assets/app.js` 第 1613-1615 行），目前每次 `render()` 都是把 `contentGrid` 整個 `innerHTML` 重新蓋一次（`sections.map(renderSection).join("")`），不是局部更新。這個架構下：

- **獨立詳情頁**：這個專案沒有路由（沒有 URL 切換、沒有 history API），`state.page` 只是內部字串切換，硬要做「詳情頁」等於要再造一個跟現有 `nav`/`page` 平行的機制，複雜度不成比例，不建議。
- **Inline row action（表格內直接編輯）**：在目前「整個表格用字串樣板 `join("")` 拼出來」的寫法下，要做到「某一列被點擊後變成可編輯的 input，其他列維持顯示」需要額外的 per-row 狀態追蹤跟局部 DOM 操作，這跟現有「整區塊重繪」的簡單模式不搭，會讓程式碼複雜度明顯上升。
- **Modal**：不管在哪個頁面點擊「新增」或某一列的「編輯」，都是同一個浮層元件、同一套邏輯（開啟時填入初始值、送出後呼叫 API、成功後關閉+重新讀取）。跟現有「整區塊重繪」的架構完全相容——modal 開著的時候 `contentGrid` 要重繪也不會互相干擾，只要 modal 本身是掛在 `contentGrid` 外面的獨立元素。

**實作上有兩個目前架構還沒有的東西，需要一起補**：

1. `index.html` 需要新增一個持久存在的 modal 容器（比照現有 `#loginScreen`/`#appShell` 這種「獨立於 `contentGrid` 之外」的做法，例如 `#formModal`），不能讓 modal 內容混在 `renderSections()` 每次重繪的範圍裡，否則 `contentGrid` 一重繪，開著的 modal 會被沖掉。
2. 目前 `renderSection()` 的 `table`/`list` 兩種類型都**沒有「每列一個操作按鈕」的欄位**——業務需求單表格現在只有「需求/提出人/類型/優先級/狀態」五欄，沒有操作欄；待決策中心的 `list` 型別也沒有操作按鈕。這兩個 render 函式需要擴充成可以帶一個可選的「操作」欄位/按鈕，這是 Batch 6A 的必要前置工程，不是可以跳過的細節。

## 3. 每個操作應寫入哪些欄位，是否有缺欄位

### 操作 1：業務提出素材需求 → `sales_requests`

你列的欄位：`request_name`、`request_type`、`priority`、`due_date`、`description`、`requested_by`。

- **`requested_by` 不應該是表單欄位，應該是登入狀態直接帶入**：`state.auth.email`（已經在 `state` 裡，Batch 1 就在用），不要讓使用者自己打信箱——這樣才能保證這筆需求真的是登入者本人送出的，不會被打錯或誤填成別人。
- **少了 `lead_id`**：原始 wireframe（p.14）明確列過「關聯案件：可選擇名單、標案、客戶或公會活動」，`sales_requests.lead_id` 這個欄位 Batch 4 就已經建好了。目前表單欄位清單沒有它——如果是刻意先簡化掉，之後要花額外一批才能補；如果只是這次列清單時漏掉，建議現在就加一個可選的「關聯名單」下拉選單（資料來源是已經接好的 `state.data.leads`），不會增加太多工作量，但少了它，需求單跟商機/名單模組就完全斷開，之後總經理想看「這個需求是為了哪個客戶」會找不到關聯。
- `status`、`assigned_to`、`completed_at`、`deliverable_resource_id` 正確地沒有出現在建立表單裡（這些是行銷總監處理階段才會填的欄位），這個範圍界線抓得對。

### 操作 2：行銷總監更新業務需求單 → `sales_requests`

你列的欄位：`status`、`assigned_to`、`due_date`、`description`。

- **少了 `completed_at` 的自動處理**：這個欄位在 Batch 4 就建好了（`timestamptz`，nullable），是原本設計要拿來算「平均處理天數」這種指標用的（wireframe 裡業務需求單頁本來就有「平均回覆：2.5天」這個 KPI）。**這個欄位沒有資料庫預設值，不會自動產生**——建議在狀態改成「已完成」的當下，前端程式碼順手把 `completed_at` 設成當下時間一起送出；如果狀態從「已完成」被改回其他狀態（例如打錯又改回處理中），`completed_at` 建議一併清成 `null`，避免留下不準確的完成時間。這件事不做，`completed_at` 會永遠是空的，將來想做「需求平均處理天數」這種報表會沒有資料可以算。
- `priority` 沒有列在行銷總監可以改的欄位裡——這是合理的範圍界線（優先級由提出人決定，行銷總監如果不同意用討論的，不需要在系統裡改），不需要調整，這裡只是確認一下你的範圍設計是刻意的還是漏掉的。

### 操作 3：總經理核准 / 需修正 → `approval_requests`

你列的欄位：`status`、`decision_note`、`decided_by`（自動）、`decided_at`（自動）。

- **`decided_by`、`decided_at` 不是資料庫自動寫入，是前端要自己組值**：這兩個欄位都沒有資料庫層的預設值或觸發器（`decided_at` 甚至連 `default now()` 都沒有），「自動寫入」指的應該是「前端邏輯自動帶入，使用者不用手動填」，不是資料庫會自己算——這點如果理解成後者，Codex 寫 SQL 時可能會去找觸發器怎麼設，其實根本不需要，直接在 PATCH 的 request body 裡帶 `decided_by: state.auth.email`、`decided_at: new Date().toISOString()` 就好。
- **`updated_at` 也要自己帶**：`sales_requests`/`approval_requests` 這兩張表跟這整個專案所有表一樣，`updated_at` 只有 `insert` 時的預設值，沒有「更新時自動變動」的觸發器（整個 v1/v2 repo 檢查下來都沒有這種 trigger 慣例）。操作 2 跟操作 3 這兩個 UPDATE 動作，都要記得在送出的 payload 裡加一行 `updated_at: new Date().toISOString()`，不然這個欄位會永遠停在建立當下的時間，之後想排序「最近更新的需求/審核」會不準。

## 4. 是否需要新增 SQL 欄位或表

**不需要**。三個操作要寫入/更新的欄位，`sales_requests`／`approval_requests` 在 Batch 4 建表時就已經全部具備，這是這次審查裡少數不用碰 SQL 的一批（前五批都或多或少動到 schema）。唯一可能需要的 SQL 異動，是第 1 節建議的選項 A（廠商報價待核准時順手 `insert` 一筆 `approval_requests`）——但那也是應用層程式碼呼叫 `INSERT`，不是新的表或欄位，一樣不用寫 migration。

## 5. 角色權限：Phase 1 先用前端控制，還是某些操作需要立即加 RLS

延續原本的決策（Phase 2 才做資料庫層權限），**但這次要點出一個風險層級的變化**：前五批全部是「讀取」，最壞情況是某個角色看到不該看的資料；Batch 6A 第一次碰「寫入」，最壞情況變成**某個角色可以做不該做的操作**——例如業務直接用瀏覽器開發工具打 API，把自己的需求單狀態改成「已完成」，或更嚴重的，`approval_requests` 目前的 grant 是「任何登入帳號都能 `insert`/`update`」，理論上業務可以直接 PATCH 一筆審核紀錄，填自己的信箱進 `decided_by`、把 `status` 改成「已核准」，繞過總經理，UI 上雖然只有總經理角色看得到核准按鈕，但 API 層沒有任何東西真的擋住其他角色這樣做。

**這不是說现在就一定要加 RLS**——公司內部使用者少、彼此信任，這個風險目前發生機率低。但因為這是「業務行動」被繞過（不是「看到不該看的資料」），風險性質不一樣，值得你明確決定一次，而不是照抄前五批「反正都先不做」的預設：

- **選項一**：維持現狀，Phase 1 全部先不管，等 Phase 2 一次做完整的 RLS。
- **選項二**：只針對 `approval_requests` 這一張表提前做一個小範圍的 RLS（只允許 `decided_by`/`status` 欄位被角色等於 `approver_role` 的登入者更新），因為這張表牽涉金額核准，風險等級跟其他表不太一樣，值得提前处理，也能順便當作 Phase 2 全面 RLS 的一個範本先驗證做法可行。

兩個都合理，這是取捨題不是對錯題，需要你決定（第 7 節）。

## 6. 是否會影響舊平台或既有資料

**不會，而且這次風險比前五批都低**。Batch 6A 不需要任何 SQL migration（第 4 節），純粹是 `assets/app.js`（加表單/modal/操作邏輯）跟 `core/api.js`（補 `POST`/`PATCH` helper，目前只有 `GET`）的應用層程式碼異動。不涉及任何資料表結構變更，對 v1 舊平台完全零接觸，也不會動到既有資料——唯一會「新增資料」的地方是使用者實際操作後產生的新列（新的需求單、新的審核決定），這是功能本身要做的事，不是意外的資料異動。
