# Claude Code Review｜Phase 1 Batch 7 草案：已取消紀錄查詢 + 手機優先原則文件化（寫程式前審查）

審查日期：2026-07-16
審查對象：已取消廠商合作/交付物查詢、已取消業務需求單查詢、`docs/phase-1-mvp.md` 手機優先原則、手機版驗收清單
審查方式：重新核對 `loadCampaignVendors()`/`activeCampaignVendors()`（Batch 6C 實際程式碼）、`loadSalesRequests()`（Batch 6A 實際程式碼）目前抓資料跟過濾的方式，確認這批要新增的「已取消」查詢實際上該怎麼接，不是憑空假設。

---

## 建議 Batch 7 範圍

四項都可以放進同一批，複雜度都不高，而且彼此獨立不互相依賴：

1. 手機優先原則寫進 `docs/phase-1-mvp.md`。
2. 廠商合作/交付物頁加「已取消紀錄」，只讀。
3. 業務需求單頁加「已取消需求」，只讀。
4. 建立手機版驗收清單文件。

**建議把第 4 項（驗收清單）排在最前面先做**，理由：清單做好之後，剩下三項（尤其第 2、3 項有實際 UI 產出）就可以邊做邊比對，不用等全部做完才回頭補驗收，這樣可以確保這批做出來的「已取消紀錄」畫面從一開始就是照手機優先的標準蓋的，而不是先做完桌機版再補手機樣式。

---

## 不建議放入 Batch 7 的項目

1. **恢復（復原已取消）功能**——你自己提案就明確排除了，這個判斷是對的，理由見下方「是否要先做只讀」那節，這裡先確認維持排除。
2. **幫其他還沒有取消機制的表（例如 `leads`、`product_knowledge_items`）補上 `cancelled_at` 之類的欄位**——這批的標題是「已取消紀錄查詢」，範圍應該限定在**已經有** `cancelled_at` 欄位的兩組資料（廠商合作/交付物、業務需求單），不要順便擴大到其他表。其他表要不要做軟取消是各自獨立的設計問題，之後遇到才處理。
3. **任何資料庫層的權限異動**——這批純粹是「用不同的過濾條件查詢已經開放讀取的表」，不需要動 RLS 或 grant，詳見下方權限風險那節。
4. **手機版驗收清單套用到「已經上線的舊頁面」的全面重新測試**——清單建立後應該作為**之後每一批新功能**的標準，不建議這批順便把 Batch 1-6C 已經上線的所有頁面都拿清單重新測一次，工作量不成比例，之前已經個別審查驗證過的頁面（登入畫面、交付物分區）不用重測。

---

## 是否需要 SQL

**核心功能不需要，但有一個對稱性的選擇題。**

重新核對 `loadCampaignVendors()` 目前實際的查詢字串，`select=...cancelled_at,cancelled_by,cancel_reason,...marketing_campaign_vendor_deliverables(...,cancelled_at,cancelled_by,cancel_reason)`——**已取消廠商合作跟已取消交付物需要的欄位，現在的查詢早就已經整包撈回來了**，只是 `activeCampaignVendors()` 這個前端函式把已取消的列過濾掉、沒有存進畫面會用到的 state。這代表廠商合作/交付物的「已取消紀錄」功能**完全不需要新的 API 查詢，也不需要任何 SQL**，純粹是前端要保留一份「已取消」的資料、再多寫一個渲染區塊。

業務需求單這邊不太一樣：`loadSalesRequests()` 目前是**直接在 URL 上加 `&cancelled_at=is.null`**，用資料庫層過濾，已取消的需求根本沒有被撈下來過。要顯示「已取消需求」，需要**多一次 API 查詢**（`&cancelled_at=not.is.null`），這不是 SQL 異動，是前端多加一個 `safeGET` 呼叫，一樣不需要動資料庫結構。

**唯一值得考慮、但不強制的 SQL**：`sales_requests` 目前**沒有** `cancel_reason` 欄位（跟 `marketing_campaign_vendors`/`marketing_campaign_vendor_deliverables` 不對稱，那兩張表有），所以已取消需求的畫面能顯示「誰、什麼時候取消」，但顯示不出「為什麼取消」。要不要現在補一個 `cancel_reason text`（nullable，純新增，不影響既有資料）讓三種已取消紀錄呈現方式一致，這是需要你決定的地方（列在最後一節）。不補也不影響這批能不能做，只是顯示內容會少一欄。

---

## 已取消廠商合作與已取消交付物：查詢與呈現建議

**不用新增查詢，改成保留兩份 state**：目前 `loadExistingData()` 只把 `activeCampaignVendors(campaignVendors)` 過濾後的結果存進 `state.data.campaignVendors`，原始（含已取消）的陣列沒有被留下來。建議改成同時保留兩份——`state.data.campaignVendors`（現行的、只有進行中的）不變，另外加一個 `state.data.cancelledCampaignVendors`（廠商合作本身被取消的那些列）跟從所有廠商（不分廠商本身是否被取消）裡挑出**交付物層級**被取消的項目彙整成 `state.data.cancelledDeliverables`。兩者都是對同一份已經抓回來的原始資料做不同的前端篩選，不會多打一次 API。

**建議拆成兩個獨立區塊呈現，不要混在一起**，因為取消可能發生在兩個不同層級，混在一起使用者會分不清楚：

- **已取消的廠商合作**：整筆 `marketing_campaign_vendors` 被取消的（不管底下交付物有沒有各自被取消），顯示廠商名稱、專案、取消原因、取消人、取消時間。
- **已取消的交付物**：`marketing_campaign_vendor_deliverables` 被取消的，**不限定廠商本身是否還在進行中**（一個廠商合作繼續進行，但其中一筆交付物被取消換掉，這種情況也要顯示出來），顯示交付物名稱、所屬廠商、取消原因、取消人、取消時間。

**呈現位置建議**：不要做成永遠顯示在畫面上的第三個表格（廠商合作頁現在已經有兩個區塊，資料量一多手機版會更擠），改成一個「查看已取消紀錄」的按鈕/切換開關，點了才展開已取消清單，預設收起。這樣平常打開頁面看到的還是乾淨的進行中清單，需要回顧歷史才多按一步。

## 已取消業務需求單：查詢與呈現建議

**需要新增一次查詢**（`sales_requests?...&cancelled_at=not.is.null`），跟現行 `loadSalesRequests()` 抓法一致，只是條件相反。**角色範圍要延續現有的既定規則**：業務登入只看得到自己取消過的需求（沿用 `visibleSalesRequests()` 已經確立的「業務只看自己的」邏輯，不要在已取消清單這裡又開一個例外讓業務看到別人的），行銷總監可以看到全部人取消過的需求。

呈現方式跟廠商合作那邊一樣，建議做成收合的「已取消紀錄」區塊，不要預設展開。

---

## 是否要先做只讀歷史紀錄，不做恢復功能

**同意你的判斷，這批應該只做只讀。** 「恢復」聽起來像是取消的反向操作，但實際上牽涉到的判斷比想像中多：

- 恢復一筆已取消的廠商合作，要不要連動把 Batch 6C 那次自動關閉的 `approval_requests`（狀態被改成「需修正」）也一起處理？還是維持那筆歷史紀錄不變，恢復後重新走一次送審？
- 恢復一筆廠商合作時，如果它底下有交付物在取消當下也各自被取消了，恢復廠商合作要不要連帶問使用者「要不要也恢復這些交付物」？
- 恢復需求單時，如果原本已經有 `assigned_to`/`completed_at` 之類的欄位在取消前就有值，恢復後這些欄位該維持原狀還是重置？

這些都是需要重新設計一次流程才能回答的問題，不是「加一個恢復按鈕」這麼單純，跟「只讀查詢」的複雜度不是同一個量級，這批不做是對的決定，之後真的有人需要復原功能，應該另外開一輪設計討論。

## 手機優先驗收清單：建議放哪份文件、包含哪些標準

**建議獨立開一個新文件**（例如 `docs/MOBILE_ACCEPTANCE_CHECKLIST.md`），不要塞進 `docs/phase-1-mvp.md`。理由：`phase-1-mvp.md` 是一份隨著每個 batch 演進、記錄「做了什麼決定」的規劃文件；驗收清單的性質不一樣，是**之後每一批新功能都要重複拿出來對照**的固定標準，混在會一直變動的規劃文件裡，之後要找會越來越不方便。`phase-1-mvp.md` 這邊只需要加一句話講清楚「手機優先是產品原則」，然後連結到這份獨立的清單文件。

**建議的最低標準**（根據這系列審查目前為止實際抓到的問題整理，不是憑空列的）：

1. **375px 寬度下沒有橫向捲動，文字不會被裁切或跟其他元素重疊**（對應這次抓到的登入畫面白字疊字問題，雖然那次成因是顏色不是裁切，但同一類「內容看不完整」的問題都該用這條檢查）。
2. **所有可點擊元件（按鈕、輸入框、下拉選單）的觸控範圍至少 44×44px**。
3. **同一張卡片/同一列裡如果有多組操作按鈕（例如巢狀的子項目各自有編輯/取消），要有清楚的視覺分區，不能讓使用者分不清楚按下去影響的是哪一筆資料**（對應這次抓到的廠商合作/交付物按鈕混淆問題，這是這系列審查目前找到最實質的一個手機版風險，值得寫進清單長期把關）。
4. **Modal 表單在手機視窗高度下要能完整捲動填寫，送出/取消按鈕不能被截斷或蓋住**。
5. **空狀態、載入中、示範資料三種狀態的文案在手機版一樣要完整顯示**，不因版面壓縮而被截斷或消失。
6. **文字顏色在其背景色上要清楚可讀**——這條直接對應登入畫面那次的根本原因（共用元件在不同背景色情境下顏色被覆蓋掉），建議明確列成一條檢查項，不要只歸類在「裁切」底下，因為成因不同、以後要抓的方法也不同（裁切用截圖看得出來，顏色對比有時候要用 `getComputedStyle` 才抓得到，跟這次的做法一樣）。
7. **核心操作（新增/編輯/取消/送審）至少要在手機寬度下實際走過一次流程**，不是只看列表畫面的排版，要真的點開表單、送出、確認結果。

**建議清單裡也附上怎麼測**：用本機 preview（`.claude/launch.json` 裡已經設定好的 `meisun_marketing_os_v2`）+ Claude Browser 工具的 `resize_window`（375×812）+ 截圖；沒有登入憑證要測某個渲染函式排版時，可以用 `javascript_tool` 直接呼叫該函式產生 HTML、注入頁面截圖驗證（這次驗證交付物卡片分區用的就是這個方法）。把這個測試方法寫進清單，之後不管是誰接手審查都能照做，不用每次重新摸索。

## 是否有權限或資料洩漏風險

**沒有新增風險。** 這批要查的兩組「已取消」資料，本來就在已經開放 `select` 給 `authenticated` 的表上（`marketing_campaign_vendors`/`marketing_campaign_vendor_deliverables`/`sales_requests`），從來沒有因為某一列被標記「已取消」就被資料庫層擋住讀取——現況其實是任何登入帳號**現在就已經技術上查得到**已取消的資料（例如直接打 `GET .../sales_requests?cancelled_at=not.is.null`），只是目前沒有任何畫面把這些資料顯示出來。這批要做的只是**把本來就查得到的資料，做一個正式的畫面呈現出來**，不是開放新的存取權限，風險沒有變化。

唯一要維持的是**角色範圍的一致性**（業務只看自己的已取消需求，不看別人的），這點延續既有邏輯即可，不是新的風險，是延續舊有正確做法。

## 是否會影響 v1 或既有資料

**不會。** 這批完全不需要新的 SQL（除非你決定加 `sales_requests.cancel_reason`，那也是單一個 nullable 欄位的純新增），純粹是前端多留一份資料、多兩個渲染區塊、多一次已經存在的表的查詢（換個過濾條件）。不會動到任何 v1 也在用的表，跟 v1 完全無交集。

## 前端實作建議

1. `loadExistingData()`：`loadCampaignVendors()` 抓回來的原始資料同時產生兩份——`activeCampaignVendors(...)` 存進現有的 `state.data.campaignVendors`（不變），新增 `cancelledCampaignVendors(...)`/`cancelledDeliverablesFromAll(...)` 兩個對稱的篩選函式，存進新的 state 欄位。
2. `loadSalesRequests()` 旁邊新增 `loadCancelledSalesRequests()`，查詢條件相反（`cancelled_at=not.is.null`），業務角色一樣要帶 `requested_by` 過濾。
3. 兩個頁面（廠商合作/交付物、業務需求單）各自加一個「查看已取消紀錄」的收合區塊，預設收起，展開後顯示對應的已取消清單，比照現有 `type:"table"` 或 `type:"list"` 的渲染方式即可，不需要新的元件類型。
4. 記得套用第「手機優先驗收清單」節列的標準去檢查這兩個新區塊，尤其是展開/收合按鈕本身的觸控大小、清單項目在手機版轉卡片後是否清楚。

## 驗收方式

1. **廠商合作**：取消一筆廠商合作、取消一筆交付物（母廠商合作維持進行中），確認「已取消紀錄」區塊各自正確顯示這兩筆，且進行中的清單完全看不到它們（延續 Batch 6C 已經驗證過的行為）。
2. **業務需求單**：業務帳號取消自己的一筆需求，確認自己看得到；確認另一個業務帳號登入後在已取消清單裡看不到別人取消的需求；行銷總監登入應該兩筆都看得到。
3. **手機版**：用 375×812 展開兩個「已取消紀錄」區塊，對照驗收清單逐條檢查（觸控大小、按鈕分區、文字完整顯示）。
4.（如果決定補 `sales_requests.cancel_reason`）確認舊的已取消需求（這個欄位加入之前就已經被取消的）顯示成「未填寫原因」或類似的空值提示，不要顯示成 `null` 字樣或報錯。

## 阻塞問題

無。這批不依賴任何還沒完成的前置工作，四個項目都可以立刻開始。唯一需要你在動工前先回答的是「`sales_requests.cancel_reason` 要不要補」這一題（前面已經列過），這不是阻塞，只是會影響業務需求單那邊已取消清單要不要顯示原因欄位。
