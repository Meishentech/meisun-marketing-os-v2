## Soft Cancel 欄位與查詢排除邏輯

**正確**。`cancelled_at`/`cancelled_by`/`cancel_reason` 三個欄位都正確加在 `marketing_campaign_vendors` 跟 `marketing_campaign_vendor_deliverables` 兩張表，型別（`timestamptz`/`citext references app_user_access(email)`/`text`）都對。

查詢排除邏輯用的是**前端過濾**而不是 URL 查詢字串過濾（不同於 `sales_requests` 用 `&cancelled_at=is.null` 直接寫在 REST 查詢裡）：`loadCampaignVendors()` 把資料連同 `cancelled_at` 一起抓回來（含巢狀的交付物），再由 `activeCampaignVendors()` 在前端把已取消的廠商列、以及每個廠商底下已取消的交付物都濾掉，才存進 `state.data.campaignVendors`。**這個做法是對的**，因為這裡的查詢同時要處理「巢狀資料」（廠商 + 交付物一次撈）跟「新舊 schema 相容」（下面會提到的 fallback 機制），用前端過濾比硬湊 PostgREST 的巢狀資源過濾語法更穩妥，資料量在 Phase 1 規模下（`limit=100`）效能也不是問題。

**額外發現一個做得很好的地方**：`loadCampaignVendors()` 有兩層 fallback——先試撈含 `cancelled_at` 等新欄位的完整查詢，如果失敗（例如萬一有人在 SQL migration 還沒執行的環境開這個頁面，新欄位不存在導致查詢報錯），會自動退回撈舊版欄位組合的查詢，不會讓整個廠商頁掛掉。這跟 `sales_requests` 那邊已經驗證過的 fallback 模式一致，這次廠商合作頁也照做了，是穩健的工程習慣。

## 取消廠商合作是否正確處理 `approval_requests` 待審核資料

**正確，而且處理得比原本建議的更完整**。`closePendingVendorApprovals()` 會在取消廠商合作的同一個操作裡，把該廠商所有 `entity_type='vendor_quote'` 且 `status='待審核'` 的審核紀錄，一併改成 `狀態=需修正`、`decided_by`/`decided_at` 自動帶入、`decision_note` 自動寫入「廠商合作已取消：{取消原因}」——不需要總經理事後自己發現一筆過時的待審項目。

**確認了一個容易誤解的細節**：`closePendingVendorApprovals()` 只處理 `status==='待審核'` 的審核紀錄，**已經核准（`已核准`）的審核紀錄不會被動到**——這是對的行為，一筆已經核准過的報價，之後即使廠商合作被取消，那筆「當初核准過」的歷史紀錄應該維持原樣，不應該被追溯性地改成「需修正」，取消是取消，核准歷史是核准歷史，兩者不衝突。

## 建議立即修正

**取消廠商合作的確認文案可以更精確**：目前寫「目前關聯交付物：N 筆。交付物資料會一起保留。」——這句話讓人以為交付物之後還能單獨查到，但實際上交付物是巢狀顯示在廠商列底下的，廠商列一旦被取消就整列從清單消失，交付物雖然真的還留在資料庫裡（沒有被砍掉），但目前沒有任何畫面能單獨看到它們。建議把文案改成類似「交付物資料會一起保留，但取消後不會顯示在任何清單中」，讓使用者的預期跟系統實際行為一致，避免之後有人以為可以找回來卻找不到而困惑。這是一行文字的修改，成本很低。
